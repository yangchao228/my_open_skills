## 正文

一篇长文写完后，真正的分发工作才刚刚开始。

公众号需要完整叙事，知乎需要问题意识和论证空间，小红书需要具体场景、首屏钩子和卡片结构。内容改完以后，还要逐张处理图片：有些图片仍然引用电脑里的本地路径，有些已经上传到某个平台，有些链接过一段时间就无法访问。

同一篇内容分发到三个平台，很容易同时出现三份文案、三套图片和三种维护方式。下一次修改时，你很难确认哪一份才是最新版本。

过去一段时间，我一直在整理自己的 AI 工作流，并把其中适合公开复用的部分放进 `my_open_skills`。这次我想做一次完整测试：拿一篇已经写完的公开长文作为输入，看看这些 Skills 能否把它整理成一套可检查、可复用的多平台内容资产。

## 我把分发流程拆成了 7 个 Skills

原案例保留了 7 个 Skills 的历史运行记录。根据这次回归得到的 V2 流程，长文插图被独立成 `article-to-illustrations`，公众号与知乎共用事实和视觉母版，小红书继续走卡片分支。

后续补充的 publish-ready 卡片合同会把卡片分支继续交给 `cards-to-images`，直到生成真实图片、完成逐图检查并留下 manifest。下图保留当时的 V2 长文插图回归状态，最新卡片交接见 [完整回归记录](workflow-v2-regression.md)。

![审核后的母稿经过发布预检、长文插图、小红书卡片、R2 图片计划和最终人工确认](https://images.reai.group/my-open-skills/theme-01/longform-v2/bc8d4f1ba0353efe-01-content-flow-wechat-16x9.png)

这次的真实输入，是一篇已经公开的《Loop Engineering 从入门到进阶手册》介绍稿。它在案例里只承担输入作用，整条链路要验证的是 Skills 技能包。

我把工作拆成了 7 段：

| 阶段 | Skill | 负责的事情 |
| --- | --- | --- |
| 源文诊断 | `wenchang-review` | 找出核心判断、真实案例、待核事实和需要放弃的材料 |
| 知乎入口 | `zhihu-topic-hunter` | 把长文转成有讨论空间的问题与论证结构 |
| 小红书入口 | `xiaohongshu-topic-generator` | 把主题转成具体场景、封面钩子和卡片叙事 |
| 通用卡片化 | `long-to-cards` | 让每张卡只承担一个判断，并补视觉方向 |
| 公众号分发 | `wechat-to-cards` | 生成文章贴图、摘要、朋友圈与社群分发资产 |
| 图片交付 | `md-img-r2` | 将本地图片整理成可审查的 R2 公网引用 |
| 发布检查 | `wenchang-publish-check` | 汇总标题、摘要、标签、链接、CTA 和图片验收条件 |

每个 Skill 只处理一个边界清楚的阶段。源文诊断有问题，就回到诊断层；小红书结构不成立，就只调整小红书入口；图片路径不稳定，就交给图片交付层处理。

这种拆法带来的直接收益，是中间判断能够被保存。下一篇文章不需要从头解释所有要求，也能替换其中某个环节。

## 内容适配解决“怎么讲”

同一篇长文进入不同平台时，我保留的是事实、核心判断和真实案例，重新设计的是读者入口。

知乎版本会先提出一个有推理空间的问题，再用案例和边界展开；小红书版本从具体痛点进入，把完整论证压缩成 8 页可保存的卡片；公众号承担完整过程叙事，让读者理解我为什么要做这套 Skills，以及每个环节如何连接。

这一步的重点在取舍。平台适配不会简单缩短字数，也不会把同一个标题换三种说法。它要回答：这个平台的读者为什么会点开？看完能带走什么？作者手里的哪份材料能支撑这个结论？

内容适配完成以后，我还遇到一个更基础的问题：同一套图片怎样在多个平台之间稳定复用？

## 图片资产，才是多平台分发里容易丢失的一层

![本地 Markdown 图片经过 dry-run 和人工审核后进入 R2，再作为多平台源图片资产](https://images.reai.group/my-open-skills/theme-01/longform-v2/c04e5e3564c94346-03-r2-flow-wechat-16x9.png)

Markdown 里的本地图片路径，通常只在作者自己的电脑上成立。

文章一旦移动目录、复制到另一台设备、进入 GitHub，或者交给另一个 Agent 继续处理，图片引用就可能失效。直接在每个平台重复上传也会产生新的问题：同一张图出现多个副本，更新时无法确认源文件，后续做博客、知乎或公众号分发时还要重新找图。

我为此做了 `md-img-r2`。

它把图片交付拆成一条可审查的路径：

```text
Markdown 本地图片
        ↓
先生成 dry-run 图片计划
        ↓
人工检查文件、对象名称和目标公网地址
        ↓
确认后上传到 Cloudflare R2
        ↓
把 Markdown 引用替换为稳定公网 URL
        ↓
        公众号 / 知乎长文 / 博客共用稳定公网引用
```

这里有一个我很看重的设计：默认运行只生成计划，不上传图片，也不改写 Markdown。只有确认目标路径和公开 URL 后，才会进入上传或写入阶段。实际改写时还会保留备份和结果报告。

这样做可以把本地文件变成有稳定地址的数字资产。文章搬到其他目录、公开到 GitHub、交给新的工作流时，图片仍然能沿用同一个公网引用。

这就是多平台分发里的一个巧思：内容层负责为不同平台重新组织表达，R2 负责保存图片源资产，`md-img-r2` 负责连接 Markdown 本地图片与公网引用。

R2 解决的是长文 Markdown 图片的公网引用问题。本次公众号和知乎长文插图进入 R2；公众号贴图、小红书图文和知乎想法配图保留本地 PNG，在平台编辑器中人工上传。

## 这次图片检查得到一个“空结果”

本次源文里有 3 张图片，它们已经通过 R2 公网域名引用。

因此，这轮 `md-img-r2` 检查得到的结果是：0 张本地图片需要上传，3 张公网图片可以继续引用，没有发生重复上传，也没有改写源文。

这个结果看起来没有动作，却说明图片资产层已经具备跨平台复用条件。工作流应该允许安全跳过已完成的阶段，同时把检查状态记录下来。

我也不准备把“没有上传”包装成一次技术成果。它真正证明的是：这条链路既能处理本地图片，也能识别已经具备公网地址的图片，并在没有必要时停下来。

## 为什么要把它做成 Skill

如果这只是一个上传脚本，它只能解决“把文件放进 R2”这一步。

我更希望它参与完整的内容交付流程：先识别 Markdown 里的本地图片，生成计划，等待人确认，上传后替换引用，再把结果交给发布检查。任何文件缺失、超出项目范围或公网地址不可信时，流程都应该停下。

这也是我开源 Skills 时反复保留的原则：

- 默认先检查和生成计划
- 外部写入需要明确确认
- 中间状态可以被人查看和修改
- 失败时说明原因、影响和下一步
- 账号登录、平台发布和最终内容取舍保留人工判断

Skill 的价值来自稳定的判断过程。脚本只是其中一个执行部件。

## 你可以怎样复现

最简单的方式，是从一篇你已经公开或获得授权的长文开始：

```text
源文诊断
  → 知乎与小红书入口
  → 卡片和分发资产
  → 图片 R2 状态
  → 发布前检查
```

如果图片仍然是本地路径，先让 `md-img-r2` 生成 dry-run 计划，检查无误后再决定是否上传。若图片已经是稳定公网 URL，就记录状态并跳过上传。

内容工作流和图片资产层都可以被复用，最终取舍仍然由你完成。

## 结尾与讨论引导

多平台内容分发的价值，最终会落到一套可以长期维护的个人系统上：观点有源文，平台稿有各自结构，图片有稳定地址，发布前有人做最终判断。

当这些过程能够被记录、复用和公开讨论，一篇文章就会继续生长为模板、案例、Skill 和后续内容。

你在多平台分发里最容易卡住的是哪一步：内容改写、图片处理、平台格式，还是发布前检查？
