# 我亲手打造的 Skills：知乎 × 小红书公开系列

## 系列定位

这个系列公开记录一组可复用 AI Agent Skills 的真实打造、使用和取舍过程。

- 系列名：`我亲手打造的 Skills`
- 公共仓库：`My Open Skills`
- 内容工作流子品牌：`Wenchang / 文昌`
- 核心读者：已经使用 AI，希望把重复工作沉淀成稳定流程的个人创作者、开发者和小团队
- 第一季目标：让读者完成一次可复现试用；关注、收藏和 GitHub Star 作为辅助信号

截至 2026-07-20，仓库有 20 个 `ready` skills。完整状态见 [Skill Catalog](../../catalog.md)，最终以仓库校验结果为准。

## 为什么现在做

Skills 已成为可共享、可复用的工作流载体。OpenAI 官方说明 Skills 可用于 ChatGPT、Codex 和 API，并遵循 Agent Skills 开放标准；Agent Skills 官方项目也把它定义为可被 agent 发现和使用的指令、脚本与资源目录。

- [OpenAI：Skills in ChatGPT](https://help.openai.com/en/articles/20001066)
- [Agent Skills 开放格式](https://agentskills.io/)

这个系列只使用这些事实作为生态背景，不把外部关注度转换成对本仓库效果的推断。

## 内容生产合同

每一期先冻结一份 Case Pack，两平台只共用事实和案例，不共用成稿。

Case Pack 必须包含：

1. 一个公开或脱敏的真实输入
2. 一份实际运行输出
3. 三个关键设计判断
4. 一个失败边界或不适用场景
5. 一条可复现路径
6. 来源日期与待核验声明

### 知乎版本

- 主要回答“为什么这样设计、解决了什么判断问题”
- 使用问题型、判断型或经验型结构
- 建议长度 1500-2500 字
- 保留证据、反例、边界和复现步骤
- 每篇只保留一个 CTA：复现、讨论或反馈

### 小红书版本

- 主要回答“在什么场景使用、输入后得到什么”
- 默认 8 页、3:4 竖版图文
- 一页只表达一个核心信息
- 必须出现真实输入、关键输出、使用步骤和不适用情况
- 系列名放在固定角标，标题优先表达读者问题
- 每篇只保留一个 CTA：收藏检查表、评论测试主题或进入仓库复现

## 第一季：主题化内容资产链路

内容按真实问题聚合，skill 名称在案例、流程图和复现入口出现。固定节奏仍为：周一冻结 Case Pack，周三发布知乎，周五发布小红书，周末记录反馈。

### 主题 01：一篇长文怎样成为跨平台内容资产

源材料：一篇已经公开或得到作者授权的长文。第一份 Case Pack 使用《Loop Engineering 从入门到进阶手册》公开介绍稿，详见 [主题 01 案例](themes/01-content-asset-pipeline/case-pack.md)。

| 环节 | 参与 skills | 对外展示的判断 |
| --- | --- | --- |
| 源文诊断 | `wenchang-review`、`storm-research` | 哪些观点、案例与事实值得保留，哪些字段需要复核 |
| 平台切入 | `zhihu-topic-hunter`、`xiaohongshu-topic-generator` | 同一长文在知乎和小红书分别从什么问题进入 |
| 发布预检 | `wenchang-publish-check` | 一次生成三平台标题、摘要、正文描述、标签时效与插图判断 |
| 平台适配与卡片成图 | `long-to-cards`、`cards-to-images`、按需使用 `wechat-to-cards` | 原文如何成为小红书、公众号贴图、知乎想法或通用卡片，并继续生成可确认的真实图片 |
| 长文插图 | `article-to-illustrations` | 公众号和知乎是否需要配图，怎样绑定正文锚点、事实约束和人工确认 |
| 图片交付 | `md-img-r2` | 本地图片插入后怎样先 dry-run，再确认上传、替换和 URL 验收 |
| 最终门禁 | `wenchang-publish-check`、`wenchang-orchestrator` | 事实、视觉、标签时效、链接和最终人工确认如何收口 |

主题 01 固定产出四种资产：

1. 公众号：用完整案例解释多平台分发、Skills 分工和 R2 图片资产层。
2. 知乎：解释跨平台内容为什么需要重组判断，而不是复制粘贴。
3. 小红书：8 页展示“源文 → 卡片 → 图片状态 → 发布检查”的真实过程。
4. GitHub：公开 Case Pack，保留输入、输出、待核事实、插图计划、图片状态与复现路径。

## 首发前承接资产

首篇发布前完成这些最低条件：

- 根 README 增加系列入口、5 分钟试用路径和兼容性说明
- 首个主题有完整源文诊断、跨平台输出和图片状态记录
- 只展示已经验证过的安装或复制方式，不写统一一键安装承诺
- 根目录保留 MIT 许可证；授权之外的效果、平台收益和运行环境兼容性不做额外承诺
- 每个案例保留生成日期、运行环境和待核验项

## 衡量与复盘

第一季以完成试用反馈为主指标，并记录以下辅助信号：

- 知乎：阅读、赞同、收藏、评论中的真实问题
- 小红书：阅读、收藏、评论测试主题、主页访问
- GitHub：访客、克隆、Star、Issue 和可复现反馈

每个主题完成后复盘一次。若读者主要停留在收藏，优先修复试用入口；若出现多个真实复现案例，再扩展到工作资产和结构化评估主题。

## 内容边界

- 不编造平台热度、节省时间、涨粉或用户效果
- 不把示例输出包装成真实客户结果
- 不公开凭据、账号会话、私有路径、真实候选人信息或发布自动化细节
- 涉及近期热点时记录来源和日期；来源不可用时标注“时效未验证”
- 保留人的最终判断，尤其是选题取舍、事实确认和外部发布
