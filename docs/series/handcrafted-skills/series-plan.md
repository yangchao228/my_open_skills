# 我亲手打造的 Skills：知乎 × 小红书公开系列

## 系列定位

这个系列公开记录一组可复用 AI Agent Skills 的真实打造、使用和取舍过程。

- 系列名：`我亲手打造的 Skills`
- 公共仓库：`My Open Skills`
- 内容工作流子品牌：`Wenchang / 文昌`
- 核心读者：已经使用 AI，希望把重复工作沉淀成稳定流程的个人创作者、开发者和小团队
- 第一季目标：让读者完成一次可复现试用；关注、收藏和 GitHub Star 作为辅助信号

截至 2026-07-10，仓库有 15 个 `ready` skills，并已通过仓库校验。完整状态见 [Skill Catalog](../../catalog.md)。

## 为什么现在做

Skills 已成为可共享、可复用的工作流载体。OpenAI 官方说明 Skills 可用于 ChatGPT、Codex 和 API，并遵循 Agent Skills 开放标准；Agent Skills 官方项目也把它定义为可被 agent 发现和使用的指令、脚本与资源目录。

- [OpenAI：Skills in ChatGPT](https://help.openai.com/en/articles/20001066)
- [Agent Skills 开放格式](https://agentskills.io/)

这个系列只使用这些事实作为生态背景，不把外部关注度转换成对本仓库效果的推断。

## 内容生产合同

每一期先冻结一份 Case Pack，两平台只共用事实和案例，不共用成稿。

Case Pack 必须包含：

1. 一个公开或脱敏的真实输入
2. 一份实际运行输出
3. 三个关键设计判断
4. 一个失败边界或不适用场景
5. 一条可复现路径
6. 来源日期与待核验声明

### 知乎版本

- 主要回答“为什么这样设计、解决了什么判断问题”
- 使用问题型、判断型或经验型结构
- 建议长度 1500-2500 字
- 保留证据、反例、边界和复现步骤
- 每篇只保留一个 CTA：复现、讨论或反馈

### 小红书版本

- 主要回答“在什么场景使用、输入后得到什么”
- 默认 8 页、3:4 竖版图文
- 一页只表达一个核心信息
- 必须出现真实输入、关键输出、使用步骤和不适用情况
- 系列名放在固定角标，标题优先表达读者问题
- 每篇只保留一个 CTA：收藏检查表、评论测试主题或进入仓库复现

## 第一季：6 周内容工作流

固定节奏：周一完成 Case Pack，周三发布知乎，周五发布小红书，周末记录反馈并决定是否调整下一期。

| 周次 | 主角 Skill | 共用案例 | 知乎核心问题 | 小红书核心场景 |
| --- | --- | --- | --- | --- |
| 1 | `xiaohongshu-topic-generator` | 为本系列生成并筛选第一批选题 | 为什么平台专用的选题判断值得单独做成 Skill？ | 我用自己写的选题 Skill 给公开系列出题 |
| 2 | `zhihu-topic-hunter` | 用同一主题生成知乎候选并和第 1 周对照 | 同一主题为什么需要不同的平台判断？ | 知乎题搬到小红书前需要改掉什么？ |
| 3 | `storm-research` | 研究“重复工作为什么值得沉淀成 Skill” | 一篇有判断的文章，动笔前要补哪些反方材料？ | 我让 AI 先做五视角研究，再决定文章怎么写 |
| 4 | `long-to-cards` | 把第 3 周知乎长文转换成 8 张卡片 | 长文转卡片时，哪些信息必须保留？ | 一篇长文拆成 8 张卡片的完整过程 |
| 5 | `wenchang-review` | 对一版真实草稿先诊断，再做最小修改 | 为什么 AI 改稿前应该先判断值不值得改？ | 我给 AI 加了一道“先诊断”的编辑流程 |
| 6 | `wenchang-orchestrator` | 串起选题、研究、写作、诊文和发布检查 | 内容工作流为什么需要多个判断节点？ | 我把内容生产拆成了一组可以组合的 Skills |

`wenchang-router`、`wenchang-research`、`wenchang-wechat-writer`、`wenchang-publish-check` 和 `wechat-to-cards` 在第 6 周作为系统组件展示，不逐个拆成相似介绍。

## 首发前承接资产

首篇发布前完成这些最低条件：

- 根 README 增加系列入口、5 分钟试用路径和兼容性说明
- 首批 3 个 skill 各有完整输入、完整输出和可展示截图
- 只展示已经验证过的安装或复制方式，不写统一一键安装承诺
- 根目录保留 MIT 许可证；授权之外的效果、平台收益和运行环境兼容性不做额外承诺
- 每个案例保留生成日期、运行环境和待核验项

## 衡量与复盘

第一季以完成试用反馈为主指标，并记录以下辅助信号：

- 知乎：阅读、赞同、收藏、评论中的真实问题
- 小红书：阅读、收藏、评论测试主题、主页访问
- GitHub：访客、克隆、Star、Issue 和可复现反馈

第 3 周和第 6 周各复盘一次。若读者主要停留在收藏，优先修复试用入口；若出现多个真实复现案例，第二季进入 `project-weekly-report`、`create-plan`、`doc-coauthoring` 和 `resume-interview-generator`。

## 内容边界

- 不编造平台热度、节省时间、涨粉或用户效果
- 不把示例输出包装成真实客户结果
- 不公开凭据、账号会话、私有路径、真实候选人信息或发布自动化细节
- 涉及近期热点时记录来源和日期；来源不可用时标注“时效未验证”
- 保留人的最终判断，尤其是选题取舍、事实确认和外部发布
